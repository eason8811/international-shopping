# customerservice 领域实现计划（按接口/定时任务分阶段）

## 1. 计划目标
- 基于 `实现要求.md`、`international-shopping-customerservice-api.yaml`、`international-shopping.sql` 中 customerservice 相关定义进行落地
- 以接口和定时任务为阶段推进, 每个阶段均包含 trigger -> domain -> infrastructure 的端到端闭环, 不按分层拆阶段
- 覆盖 customerservice 全量 37 个接口与 2 个定时任务

## 2. 跨阶段统一实现约束
- 命名、包路径、分层调用关系严格遵循 `实现要求.md` 中约定
- 所有 `POST` 写接口强制接入 `Idempotency-Key`, 在 infrastructure 层落 Redis 幂等记录, 领域服务只消费幂等结果
- Controller 入参校验与 domain 业务校验分离, Controller 优先调用 `request.validate()` 或 `FieldValidateUtils`
- 所有写库操作使用 CAS 或状态机条件更新, 避免并发覆盖, 并统一锁顺序避免与 `orders`、`payment`、`shipping` 形成死锁
- 分页接口统一使用 `PageQuery`、`PageResult`
- 嵌套对象查询优先在 `Mapper.xml` 中使用联表查询, 禁止在 for 循环中逐条访问数据库
- 所有新增类、方法、字段补充完整中文 JavaDoc, 标点按实现要求统一

## 3. 分阶段计划

### 阶段 1, 用户工单基础接口
接口范围:
- `GET /users/me/tickets`, 用户侧分页查询工单列表
- `POST /users/me/tickets`, 用户侧创建工单
- `GET /users/me/tickets/{ticket_no}`, 用户侧查询工单详情
- `POST /users/me/tickets/{ticket_no}/close`, 用户侧关闭工单

关键实现点:
- 打通 `cs_ticket` 基础读写链路, 完成用户侧工单列表、详情、创建、关闭
- 创建工单依赖 `uk_ticket_open_dedupe` 与幂等键双重防重
- 关闭工单按状态机约束落 `cs_ticket_status_log`
- 列表、详情返回字段对齐 `UserTicketSummaryRespond`、`UserTicketDetailRespond`

阶段验收:
- 上述 4 个接口可联调
- 创建与关闭均具备幂等和并发安全
- 状态变更日志可回放

### 阶段 2, 用户消息与会话接口
接口范围:
- `GET /users/me/tickets/{ticket_no}/messages`, 用户侧查询工单消息列表
- `POST /users/me/tickets/{ticket_no}/messages`, 用户侧发送工单消息
- `PATCH /users/me/tickets/{ticket_no}/messages/{message_no}`, 用户侧编辑消息
- `POST /users/me/tickets/{ticket_no}/messages/{message_no}/recall`, 用户侧撤回消息
- `POST /users/me/tickets/{ticket_no}/read`, 用户侧标记工单消息已读
- `GET /users/me/tickets/{ticket_no}/status-logs`, 用户侧查询工单状态流转日志
- `GET /users/me/tickets/{ticket_no}/reships`, 用户侧查询工单下补发单关联物流单
- `POST /users/me/ws-sessions`, 用户侧创建消息实时通道会话

关键实现点:
- 打通 `cs_ticket_message`、`cs_ticket_participant`、`cs_ticket_status_log` 查询与写入
- 发送消息使用 `uk_msg_client_dedupe` + `Idempotency-Key` 防重
- 已读接口更新参与方已读位点, 并产出 WebSocket 事件数据
- 工单关联补发单与物流单使用联表查询返回
- 生成用户侧 WebSocket 会话票据, 支撑后续握手

阶段验收:
- 用户消息增删改查与已读位点可用
- 状态日志、补发物流查询可用
- 用户侧实时会话建立可联调

### 阶段 3, 管理工单核心流转接口
接口范围:
- `GET /admin/tickets`, 管理侧分页检索工单
- `GET /admin/tickets/{ticket_id}`, 管理侧查询工单详情
- `PATCH /admin/tickets/{ticket_id}`, 管理侧更新工单元数据
- `POST /admin/tickets/{ticket_id}/assign`, 管理侧指派或转派工单
- `POST /admin/tickets/{ticket_id}/status`, 管理侧更新工单状态

关键实现点:
- 管理侧检索、详情、元数据更新链路打通
- 指派操作写入 `cs_ticket_assignment_log`
- 严格实现状态机矩阵, 并限制仅 `ASSIGNEE`、`COLLABORATOR` 可变更状态, `WATCHER` 禁止变更
- `RESOLVED/REJECTED -> IN_PROGRESS` 仅允许 7 天内复开
- 状态变更统一写 `cs_ticket_status_log`

阶段验收:
- 管理工单核心 5 接口可联调
- 角色校验、状态流转、审计日志行为符合要求

### 阶段 4, 管理消息与会话接口
接口范围:
- `GET /admin/tickets/{ticket_id}/messages`, 管理侧查询工单消息列表
- `POST /admin/tickets/{ticket_id}/messages`, 管理侧发送工单消息
- `PATCH /admin/tickets/{ticket_id}/messages/{message_id}`, 管理侧编辑消息
- `POST /admin/tickets/{ticket_id}/messages/{message_id}/recall`, 管理侧撤回消息
- `POST /admin/tickets/{ticket_id}/read`, 管理侧标记工单消息已读
- `POST /admin/ws-sessions`, 管理侧创建消息实时通道会话

关键实现点:
- 复用消息域能力并补齐管理侧权限校验
- 管理侧发送消息同样执行幂等、防并发写覆盖
- 已读位点和实时事件与用户侧保持一致协议
- 生成管理侧 WebSocket 会话票据

阶段验收:
- 管理消息 6 接口可联调
- 消息编辑、撤回、已读、会话票据行为稳定

### 阶段 5, 管理参与方与审计日志接口
接口范围:
- `GET /admin/tickets/{ticket_id}/participants`, 管理侧查询工单参与方
- `POST /admin/tickets/{ticket_id}/participants`, 管理侧新增工单参与方
- `PATCH /admin/tickets/{ticket_id}/participants/{participant_id}`, 管理侧更新参与方角色
- `POST /admin/tickets/{ticket_id}/participants/{participant_id}/leave`, 管理侧将参与方设为离开
- `GET /admin/tickets/{ticket_id}/status-logs`, 管理侧查询工单状态流转日志
- `GET /admin/tickets/{ticket_id}/assignment-logs`, 管理侧查询工单指派日志

关键实现点:
- 打通参与方增改离场与查询
- 依赖 `uk_ticket_participant` 控制活跃参与方唯一性
- 角色变更与离场动作补充业务校验, 避免非法角色组合
- 状态日志、指派日志分页查询稳定输出

阶段验收:
- 参与方与审计类 6 接口可联调
- 角色维护与审计回放符合预期

### 阶段 6, 补发单全链路接口
接口范围:
- `POST /admin/tickets/{ticket_id}/reships`, 管理侧基于工单创建补发单
- `GET /admin/reships`, 管理侧分页检索补发单
- `GET /admin/reships/{reship_id}`, 管理侧查询补发单详情
- `PATCH /admin/reships/{reship_id}`, 管理侧更新补发单元数据
- `POST /admin/reships/{reship_id}/status`, 管理侧推进补发单状态
- `GET /admin/reships/{reship_id}/shipments`, 管理侧查询补发单关联物流单
- `POST /admin/reships/{reship_id}/shipments`, 管理侧绑定补发单与物流单

关键实现点:
- 打通 `aftersales_reship`、`aftersales_reship_item`、`aftersales_reship_shipment` 全流程
- 创建补发单时批量写明细, 禁止逐条写库
- 补发状态机推进使用条件更新, 防并发覆盖
- 绑定物流单遵循 `uk_reship_one_shipment` 唯一约束
- 详情接口使用联表或分组批量查询返回明细与物流单

阶段验收:
- 补发单 7 接口可联调
- 补发明细、状态、物流绑定行为正确, 并发下无重复绑定

### 阶段 7, WebSocket 连接接口
接口范围:
- `GET /`, 建立售后消息 WebSocket 连接

关键实现点:
- 接入 WebSocket 握手鉴权, 校验来自 `POST /users/me/ws-sessions` 与 `POST /admin/ws-sessions` 的会话凭据
- 统一事件封包结构, 对齐 `CsWsEventEnvelopeRespond` 等已定义 DTO
- 支持连接确认、错误帧、增量事件推送、必要的断线恢复策略

阶段验收:
- WebSocket 握手、鉴权、事件推送可联调
- 用户侧与管理侧会话均可稳定消费消息事件

### 阶段 8, 定时任务
任务范围:
- `AWAITING_USER -> CLOSED`, 3 天无新消息或状态更新自动关闭, `note=USER_TIMEOUT`, `actor_type=SCHEDULER`
- `RESOLVED -> CLOSED`, 自动关闭, `note=AUTO_CLOSE_AFTER_RESOLVE`, `actor_type=SCHEDULER`

关键实现点:
- 在 `trigger.job.customerservice` 提供独立 `**Job` 任务
- 扫描条件走 `idx_ticket_status_update`, 分批拉取, 分批更新
- 状态更新采用 CAS 条件, 防止任务与人工操作冲突
- 每次状态推进必须写入 `cs_ticket_status_log`, `source_ref` 写任务运行标识
- `RESOLVED -> CLOSED` 默认按 7 天窗口执行, 与 7 天复开规则保持一致, 关闭阈值做可配置化

阶段验收:
- 两个自动关单任务均可重复执行且幂等
- 无漏关、无误关, 日志可追溯
